\subsection*{B.1.3. Systemarchitektur: Architekturmodelle}

Architekturmodelle dienen in der Mensch-Maschine-Interaktion dazu, die Struktur eines
interaktiven Systems so zu organisieren, dass die Benutzungsschnittstelle (UI) und die
Anwendungslogik sinnvoll getrennt werden.

Man unterteilt diese Modelle grundsätzlich in drei Kategorien:

\begin{enumerate}
  \item \textbf{Schichtenmodelle} (Beispiel: Seeheim-Modell)
  \item \textbf{Objektorientierte Modelle} (Beispiel: Model-View-Controller -- MVC)
  \item \textbf{Gerätemodelle}
\end{enumerate}

% -----------------------------------------
\subsection{Das Seeheim-Modell (Schichtenmodell)}
% -----------------------------------------

Das Seeheim-Modell ist ein klassisches Beispiel für eine hierarchische Trennung in drei
Schichten.

\textbf{Die Schichten:}
\begin{itemize}
  \item \textbf{Präsentations-Schicht:} Sie ist direkt dem Benutzer zugewandt und beinhaltet
        die visuelle Darstellung sowie die physische Interaktion (Widgets wie Buttons,
        Menüs, Eingabefelder).
  \item \textbf{Dialogsteuer-Schicht:} Sie ist das „Gehirn" bzw. „Gedächtnis" des Dialogs
        und legt fest, wie das System auf Eingaben reagiert und in welchem Zustand sich
        die Interaktion befindet.
  \item \textbf{Anwendungs-Schicht:} Sie stellt die Schnittstelle zur eigentlichen Fachlogik
        dar und definiert, welche Daten und Funktionen der Anwendung für die UI
        verfügbar sind.
\end{itemize}

\textbf{Vorteile:}
\begin{itemize}
  \item \textbf{Portierbarkeit \& Wiederverwendbarkeit:} Man kann die UI (Präsentation)
        austauschen, ohne den Code der Anwendung ändern zu müssen.
  \item \textbf{Aufgabenteilung:} UI-Designer und Programmierer der Anwendungslogik können
        unabhängig voneinander arbeiten.
  \item \textbf{Rapid Prototyping:} Die Oberfläche lässt sich schnell verändern oder als
        Prototyp bauen, da sie vom Kern getrennt ist.
\end{itemize}

\textbf{Nachteile:}
\begin{itemize}
  \item \textbf{Leistungseinbußen („Sonderwege"):} Sehr schnelle grafische Ausgaben
        (z.\,B. bei Spielen oder Echtzeit-Visualisierungen) müssen die Schichten oft umgehen,
        da der Weg durch den Dialogcontroller zu langsam ist.
  \item \textbf{Fließende Grenzen:} In der Praxis ist es oft schwer, eine strikte Grenze
        zwischen der Präsentation und der Steuerung zu ziehen.
  \item \textbf{Komplexität des Kontrollflusses:} Das Modell gibt keine klaren Hinweise
        darauf, wie genau die Nachrichten zwischen den Schichten fließen
        (Prozeduraufruf vs. Nachrichtenaustausch).
\end{itemize}

% -----------------------------------------
\subsection{Model-View-Controller (MVC)}
% -----------------------------------------

Dies ist das heute am weitesten verbreitete Entwurfsmuster für moderne Benutzeroberflächen
wie Android, Java Swing oder Qt.

\textbf{Die drei Komponenten:}
\begin{itemize}
  \item \textbf{Model (Modell):} Verwaltet die zentralen Daten und die Verarbeitungslogik.
        Es informiert registrierte Beobachter (Views) über Änderungen.
  \item \textbf{View (Ansicht):} Zuständig für die grafische Darstellung. Sie fragt die Daten
        aktiv beim Model ab und reagiert auf Änderungsmitteilungen.
  \item \textbf{Controller (Steuerung):} Verarbeitet Ereignisse (Events) wie Mausklicks oder
        Tastatureingaben. Er ruft die Logik im Model auf und leitet Änderungen an die View
        weiter.
\end{itemize}

\textbf{Vorteile:}
\begin{itemize}
  \item \textbf{Mehrfache Ansichten:} Ein Model kann gleichzeitig in verschiedenen Views
        dargestellt werden (z.\,B. dieselben Daten als Tabelle und als Kreisdiagramm).
  \item \textbf{Synchronisation:} Durch den Listener-Mechanismus (Beobachter-Muster) bleiben
        alle Ansichten automatisch aktuell, wenn sich die Daten ändern.
\end{itemize}

\begin{examplebox}
  In einem Musik-Player ist das \textit{Model} die Playlist, der aktuelle Track und der
  Wiedergabestatus. Die \textit{View} zeigt Titel, Fortschrittsbalken und Album-Cover an.
  Der \textit{Controller} verarbeitet Benutzereingaben wie Play\,/\,Pause oder Weiter.
\end{examplebox}

\textbf{Beispiele für MVC-Systeme:} Java AWT/Swing, Windows-UIs, Android-Apps, Qt-Framework.

% -----------------------------------------
\subsection{Serviceorientierte Architektur (SOA) / Client-Server}
% -----------------------------------------

Dieses Modell wird als Dreiecksbeziehung dargestellt, um Dienste flexibel zu nutzen.

\textbf{Die Akteure:}
\begin{itemize}
  \item \textbf{Service Provider:} Erstellt einen Dienst und stellt ihn bereit
        (\textit{Publish}).
  \item \textbf{Service Registry:} Ein Verzeichnis, in dem der Consumer nach passenden
        Diensten suchen kann (\textit{Find}).
  \item \textbf{Service Consumer:} Findet den Dienst und bindet ihn in seine Anwendung ein
        (\textit{Bind}).
\end{itemize}

\begin{examplebox}
  \textbf{Anwendung in der Computergraphik:} Ein Anwendungsprogramm fungiert als
  \textbf{Client} und nutzt die Dienste eines Graphiksystems (z.\,B. OpenGL) als
  \textbf{Server}.
\end{examplebox}
