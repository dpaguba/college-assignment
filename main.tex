\subsection{Systemarchitektur: Architekturmodelle}
% -----------------------------------------

Architekturmodelle dienen in der Mensch-Maschine-Interaktion dazu, die Struktur eines
interaktiven Systems so zu organisieren, dass die Benutzungsschnittstelle (UI) und die
Anwendungslogik sinnvoll getrennt werden.

Man unterteilt diese Modelle grundsätzlich in drei Kategorien:

\begin{enumerate}
  \item \textbf{Schichtenmodelle} (Beispiel: Seeheim-Modell)
  \item \textbf{Objektorientierte Modelle} (Beispiel: Model-View-Controller -- MVC)
  \item \textbf{Gerätemodelle} (Beispiel: Serviceorientierte Architektur)
\end{enumerate}

% -----------------------------------------
\subsubsection{Das Seeheim-Modell (Schichtenmodell)}
% -----------------------------------------

Das Seeheim-Modell ist ein klassisches Beispiel für eine hierarchische Trennung in drei
Schichten.

\begin{figure}[H]
\centering

  % Three layer boxes

  \node[layerbox] (dial) at (3.2,  2) {\textbf{Dialogsteuer-}\\schicht};
  \node[layerbox] (app)  at (6.4,  2) {\textbf{Anwendungs-}\\schicht};

  % Benutzungsschnittstelle frame
  \begin{scope}[on background layer]
    \node[draw, dashed, inner sep=0.5cm,

  \end{scope}

  \node[layerbox, minimum width=2cm] (anw) at (9.8, 2) {\textbf{Anwendung}};

  \draw[<->] (user)  -- (pres);

  \draw[<->] (dial)  -- (app);
  \draw[<->] (app)   -- (anw);
\end{tikzpicture}
\caption{Das Seeheim-Modell}
\end{figure}

\textbf{Die Schichten:}
\begin{itemize}
  \item \textbf{Präsentations-Schicht:} Direkt dem Benutzer zugewandt; beinhaltet
        die visuelle Darstellung sowie die physische Interaktion (Widgets wie Buttons,
        Menüs, Eingabefelder).
  \item \textbf{Dialogsteuer-Schicht:} Das „Gehirn" bzw. „Gedächtnis" des Dialogs;
        legt fest, wie das System auf Eingaben reagiert und in welchem Zustand sich
        die Interaktion befindet.
  \item \textbf{Anwendungs-Schicht:} Schnittstelle zur eigentlichen Fachlogik;
        definiert, welche Daten und Funktionen der Anwendung für die UI verfügbar sind.
\end{itemize}

\textbf{Vorteile:}
\begin{itemize}
  \item \textbf{Portierbarkeit \& Wiederverwendbarkeit:} UI lässt sich austauschen,
        ohne den Anwendungscode zu ändern.
  \item \textbf{Aufgabenteilung:} UI-Designer und Entwickler der Anwendungslogik
        können unabhängig voneinander arbeiten.
  \item \textbf{Rapid Prototyping:} Die Oberfläche lässt sich schnell verändern,
        da sie vom Kern getrennt ist.
\end{itemize}

\textbf{Nachteile:}
\begin{itemize}
  \item \textbf{Leistungseinbußen („Sonderwege"):} Schnelle grafische Ausgaben
        (z.\,B. Echtzeit-Visualisierungen) müssen die Schichten oft umgehen,
        da der Weg durch den Dialogcontroller zu langsam ist.
  \item \textbf{Fließende Grenzen:} In der Praxis ist eine strikte Grenze zwischen
        Präsentation und Steuerung oft schwer zu ziehen.
  \item \textbf{Komplexität des Kontrollflusses:} Keine klaren Vorgaben, wie
        Nachrichten zwischen den Schichten fließen
        (Prozeduraufruf vs.\ Nachrichtenaustausch).
\end{itemize}

\subsubsection{Model-View-Controller (MVC)}
% -----------------------------------------

Das heute am weitesten verbreitete Entwurfsmuster für moderne Benutzeroberflächen
(Android, Java Swing, Qt).

\begin{figure}[H]
\centering
\begin{tikzpicture}[
  comp/.style={draw, minimum width=2.4cm, minimum height=0.9cm,
               align=center, font=\small\bfseries},
  lbl/.style={font=\footnotesize, align=center},
  >=Stealth
]
  \node[comp] (view)  at (-3,    0) {View};
  \node[comp] (model) at ( 3,    0) {Model};
  \node[comp] (ctrl)  at ( 0,  -3) {Controller};

  % Model → View (dashed: notify) — oben
  \draw[->, dashed, bend right=20]
    (model) to
    node[above, lbl] {benachrichtigt registrierte Views\\(Listener) bei Änderungen}
    (view);

  % View → Model (query) — unten
  \draw[->, bend right=20]
    (view) to
    node[below, lbl] {fragt Daten ab}
    (model);

  % Controller → View
  \draw[->]
    (ctrl) -- node[left, lbl] {informiert über\\Darstellungsänderung\\(z.\,B. Textselektion)} (view);

  % Controller → Model
  \draw[->]
    (ctrl) -- node[right, lbl] {ruft Verarbeitungslogik auf,\\leitet Änderungen weiter} (model);

  % Events → Controller
  \draw[->]
    (0, -4.8) -- node[right, lbl] {Events\\(z.\,B. Benutzereingabe)} (ctrl);

  % Side labels
  \node[lbl, left=0.1cm of view]  {(Graphische)\\Darstellung};
  \node[lbl, right=0.1cm of model] {Daten und\\Verarbeitungs-\\logik};
\end{tikzpicture}
\caption{Model-View-Controller-Entwurfsmuster (MVC)}
\end{figure}

\textbf{Die drei Komponenten:}
\begin{itemize}
  \item \textbf{Model:} Verwaltet Daten und Verarbeitungslogik; informiert
        registrierte Views über Änderungen.
  \item \textbf{View:} Grafische Darstellung; fragt Daten aktiv beim Model ab
        und reagiert auf Änderungsmitteilungen.
  \item \textbf{Controller:} Verarbeitet Events (Mausklicks, Tastatureingaben);
        ruft Logik im Model auf und leitet Änderungen an die View weiter.
\end{itemize}

\textbf{Vorteile:}
\begin{itemize}
  \item \textbf{Mehrfache Ansichten:} Ein Model kann gleichzeitig in verschiedenen
        Views dargestellt werden (z.\,B. als Tabelle und als Diagramm).
  \item \textbf{Synchronisation:} Durch den Listener-Mechanismus bleiben alle
        Views automatisch aktuell.
\end{itemize}

\begin{examplebox}
  In einem Musik-Player ist das \textit{Model} die Playlist, der aktuelle Track und
  der Wiedergabestatus. Die \textit{View} zeigt Titel, Fortschrittsbalken und
  Album-Cover an. Der \textit{Controller} verarbeitet Eingaben wie Play\,/\,Pause
  oder Weiter.
\end{examplebox}

\textbf{Beispiele:} Java AWT/Swing, Windows-UIs, Android-Apps, Qt-Framework.

\subsubsection{Serviceorientierte Architektur (SOA) -- Gerätemodell}
% -----------------------------------------

SOA wird als Dreiecksbeziehung dargestellt und dient in der MMI als Gerätemodell:
die Anwendung nutzt Hardware-Dienste (z.\,B. Grafik) über einen standardisierten
Service-Kanal.

\begin{figure}[H]
\centering
\begin{tikzpicture}[
  actor/.style={circle, draw, minimum size=2.2cm, align=center, font=\small\bfseries},
  lbl/.style={font=\small},
  >=Stealth
]
  \node[actor] (reg)  at ( 0,   3) {Service\\Registry};
  \node[actor] (prov) at (-3,   0) {Service\\Provider};
  \node[actor] (cons) at ( 3,   0) {Service\\Consumer};

  \draw[->] (prov) -- node[left,  lbl] {Publish} (reg);
  \draw[->] (reg)  -- node[right, lbl] {Find}    (cons);
  \draw[<->](prov) -- node[below, lbl] {Bind}    (cons);
\end{tikzpicture}
\caption{Serviceorientierte Architektur (SOA)}
\end{figure}

\textbf{Die Akteure:}
\begin{itemize}
  \item \textbf{Service Provider:} Erstellt einen Dienst und stellt ihn bereit
        (\textit{Publish}).
  \item \textbf{Service Registry:} Verzeichnis, in dem der Consumer passende
        Dienste sucht (\textit{Find}).
  \item \textbf{Service Consumer:} Findet den Dienst und bindet ihn ein
        (\textit{Bind}).
\end{itemize}

\begin{examplebox}
  In der Computergraphik fungiert das Anwendungsprogramm als \textbf{Client} und
  nutzt die Dienste eines Graphiksystems (z.\,B. OpenGL) als \textbf{Server}.
\end{examplebox}

\textbf{Vorteile:}
\begin{itemize}
  \item \textbf{Lose Kopplung:} Provider und Consumer müssen sich nicht direkt
        kennen -- die Vermittlung läuft über die Registry.
  \item \textbf{Austauschbarkeit:} Ein Dienst (z.\,B. ein Eingabegerät) kann
        gegen einen anderen ausgetauscht werden, ohne die Anwendung zu ändern.
  \item \textbf{Wiederverwendbarkeit:} Einmal registrierte Dienste können von
        mehreren Consumern gleichzeitig genutzt werden.
\end{itemize}

\textbf{Nachteile:}
\begin{itemize}
  \item \textbf{Overhead durch Registry:} Jede Kommunikation läuft über eine
        zusätzliche Vermittlungsschicht, was Latenz erzeugen kann.
  \item \textbf{Komplexität:} Das Verwalten von Diensten (Publish, Find, Bind)
        erhöht den Implementierungsaufwand gegenüber direkten Aufrufen.
  \item \textbf{Single Point of Failure:} Fällt die Registry aus, können
        Consumer keine neuen Dienste mehr finden.
\end{itemize}

\subsubsection{Die Benutzerseite -- Kognitiver Prozess des Menschen}
% - - - - - - - - - - - - - - - - - - - - -

Der Mensch durchläuft intern vier Hauptphasen:

\begin{enumerate}
  \item \textbf{Wahrnehmung (Perception):} Der Benutzer nimmt Signale der Maschine
        über seine Sinnesorgane (Augen, Ohren, Tastsinn) auf.
  \item \textbf{Bewertung (Evaluation):} Das Wahrgenommene wird mit dem ursprünglichen
        Ziel verglichen. \textit{„Ist das passiert, was ich wollte?"}
  \item \textbf{Zielsetzung (Goal):} Basierend auf der Bewertung entscheidet der
        Benutzer, was als Nächstes zu tun ist.
  \item \textbf{Planung \& Handlung (Execution):} Der Benutzer plant die physische
        Bewegung und führt sie über sein motorisches System aus.
\end{enumerate}

% - - - - - - - - - - - - - - - - - - - - -
\subsubsection{Die Computerseite -- Informationsverarbeitung des Systems}
% - - - - - - - - - - - - - - - - - - - - -

Parallel dazu verarbeitet das System die Eingaben in vier Schritten:

\begin{enumerate}
  \item \textbf{Informationsaufnahme:} Das System registriert Signale über
        \textbf{Sensoren} oder Eingabegeräte (Maus, Tastatur, Kamera, Mikrofon).
  \item \textbf{Analyse:} Das System interpretiert die Daten -- \textbf{Bedeutungsanalyse}
        oder Mustererkennung (z.\,B. „Der Klick erfolgte auf den Speichern-Button").
  \item \textbf{Zustandsübergang:} Die Programmlogik wird ausgeführt; der interne
        Zustand der Software ändert sich (z.\,B. Datei wird gespeichert, Objekt
        in einer 3D-Szene wird verschoben).
  \item \textbf{Informationsausgabe:} Das System erzeugt Feedback über \textbf{Medien}
        (Grafik auf dem Display, Ton aus Lautsprechern).
\end{enumerate}

% - - - - - - - - - - - - - - - - - - - - -
\subsubsection{Kernbegriffe der Interaktion}
% - - - - - - - - - - - - - - - - - - - - -

\begin{itemize}
  \item \textbf{Interaktionsaufgabe (Interaction Task):} Eine \textit{abstrakte}
        Aktion innerhalb einer Anwendung.\\
        \textit{Beispiele:} Objektauswahl, Text eingeben, Objekt im Raum transformieren
        (drehen / verschieben).

  \item \textbf{Interaktionstechnik (Interaction Technique):} Die \textit{konkrete
        technische Realisierung} einer Interaktionsaufgabe.\\
        \textit{Beispiele für „Objekt auswählen":}
        \begin{enumerate}
          \item Mit der Maus auf ein Icon klicken.
          \item Den Namen des Objekts per Tastatur tippen.
          \item In einer VR-Umgebung mit einem „virtuellen Laserstrahl" zeigen.
        \end{enumerate}

  \item \textbf{Interaktionsstil (Dialogstil):} Legt fest, in welcher \textit{Form}
        der Dialog zwischen Mensch und Maschine gestaltet ist (siehe B.3).

  \item \textbf{Interaktionsumgebung:} Die technische Umgebung, in der Aufgaben,
        Techniken und Stile umgesetzt werden (siehe B.4).
\end{itemize}

% - - - - - - - - - - - - - - - - - - - - -
\subsubsection{Die vier Grundkomponenten}
% - - - - - - - - - - - - - - - - - - - - -

\begin{itemize}
  \item \textbf{Modalität} (Mensch $\rightarrow$ Maschine): Typ des Ausgabesignals,
        den der \textbf{Mensch} erzeugt.\\
        \textit{Beispiele:} Sprache, Töne, Gestik, Mimik, Körperbewegungen,
        Biosignale (Gehirnströme).

  \item \textbf{Sensorik} (Aufnahme durch die Maschine): Technische Kanäle, mit
        denen das \textbf{System} Signale des Menschen aufnimmt.\\
        \textit{Beispiele:} Mikrofone, Kameras, Touchscreens, elektrochemische
        Sensoren, elektromagnetische Antennen.

  \item \textbf{Medien} (Maschine $\rightarrow$ Mensch): Typ des Ausgabesignals,
        den die \textbf{Maschine} erzeugt.\\
        \textit{Beispiele:} Text, Grafik (Standbild/Animation), Sprachsynthese,
        künstlicher oder realistischer Ton.

  \item \textbf{Sinne} (Aufnahme durch den Menschen): Menschliche Sinnesorgane,
        die Signale der Maschine registrieren.\\
        \textit{Beispiele:} Sehen (visuell), Hören (auditiv), Tasten
        (haptisch/taktil), Riechen (olfaktorisch), Schmecken (gustatorisch).
\end{itemize}

\begin{notebox}
  Schema für die Klausur:\\
  \textbf{Mensch (Modalität) $\rightarrow$ Sensorik $\rightarrow$ Rechner
  $\rightarrow$ Medium $\rightarrow$ Sinne $\rightarrow$ Mensch}
\end{notebox}

% - - - - - - - - - - - - - - - - - - - - -
\subsubsection{Kombination mehrerer Kanäle}
% - - - - - - - - - - - - - - - - - - - - -

\begin{center}
\begin{tabular}{p{2.8cm} p{5.5cm} p{5cm}}
\hline
\textbf{Begriff} & \textbf{Erklärung} & \textbf{Beispiel} \\
\hline
\textbf{Multimedial} & Mehr als ein \textbf{Medium} (Ausgabeseite des Rechners) &
  Lernsystem mit Text, Bild und Video \\[4pt]
\textbf{Multimodal} & Mehr als eine \textbf{Modalität} (Eingabeseite des Menschen) &
  Sprechen + Zeigen: „Lösche dieses Objekt" + Fingerzeig \\[4pt]
\textbf{Multicodal} & Verschiedene \textbf{Kodierungen} für dasselbe Medium &
  „STOPP" als Wort + rotes Achteck-Symbol \\
\hline
\end{tabular}
\end{center}

\textbf{Multimediale Interaktion:}
\begin{itemize}
  \item \textbf{Vorteil:} Informationen über mehrere Kanäle (Hören \& Sehen) werden
        besser aufgenommen und behalten.
  \item \textbf{Nachteil:} Gefahr der \textbf{Informationsüberlastung} -- schwierig,
        die Aufmerksamkeit auf das Wesentliche zu lenken.
\end{itemize}

\textbf{Multimodale Interaktion:}
\begin{itemize}
  \item \textbf{Vorteil:} \textbf{Höhere Robustheit} -- bei Umgebungslärm kann das
        System die Absicht durch Gestik erkennen, auch wenn die Sprache nicht
        verstanden wird.
  \item \textbf{Nachteil:} \textbf{Technische Komplexität} -- das System muss
        verschiedene Eingabeströme (z.\,B. Lippenbewegung + Ton) in Echtzeit
        synchronisieren und verschmelzen.
\end{itemize}

% -----------------------------------------
\subsection{Interaktionsstile (Dialoggestaltung)}
% -----------------------------------------

Ein \textbf{Interaktionsstil} (auch Dialogstil) legt fest, in welcher Form der Dialog
zwischen Mensch und Maschine gestaltet ist. Es werden drei Hauptstile unterschieden:

% - - - - - - - - - - - - - - - - - - - - -
\subsubsection{Sprachbasierte Interaktion}
% - - - - - - - - - - - - - - - - - - - - -

Ein- und Ausgabe erfolgen über natürliche oder künstliche Sprache.

\textbf{A. Kommandosprachen} (z.\,B. Linux Terminal, Windows CMD):
\begin{itemize}
  \item \textbf{Vorteile:} Sehr effizient für Profis; ermöglichen Skripte und Makros.
  \item \textbf{Nachteile:} Hoher Lernaufwand (Syntax muss auswendig gelernt werden);
        hohe Fehleranfälligkeit.
\end{itemize}

\textbf{B. Natürliche Sprache} (z.\,B. Siri, ChatGPT):
\begin{itemize}
  \item \textbf{Vorteile:} Geringer Lernaufwand, da Alltagssprache genutzt wird.
  \item \textbf{Nachteile:} Ineffizient durch Redundanz; Mehrdeutigkeiten (Homonyme)
        führen oft zu Fehlern im System.
\end{itemize}

% - - - - - - - - - - - - - - - - - - - - -
\subsubsection{Direkte Manipulation (DMI)}
% - - - - - - - - - - - - - - - - - - - - -

Basiert auf der visuellen Darstellung von Objekten, die mit Eingabegeräten direkt
bewegt oder verändert werden können.

\textbf{Merkmale:}
\begin{itemize}
  \item Sichtbarkeit der Objekte
  \item Schnelle, umkehrbare Aktionen
  \item Visuelle statt textuelle Befehle
\end{itemize}

\textbf{Vorteile:}
\begin{itemize}
  \item Intuitiv; Fehler sind sofort sichtbar und korrigierbar.
  \item Weitgehend sprachunabhängig.
\end{itemize}

\textbf{Nachteile:}
\begin{itemize}
  \item Geringe Präzision.
  \item Hoher Platzbedarf auf dem Bildschirm.
  \item Kein Geschwindigkeitsvorteil für Experten.
\end{itemize}

\begin{examplebox}
  \textbf{WYSIWYG} („What You See Is What You Get") wie Microsoft Word: Das Dokument
  wird direkt so dargestellt, wie es später aussieht.
\end{examplebox}

% - - - - - - - - - - - - - - - - - - - - -
\subsubsection{Gestenbasierte Interaktion}
% - - - - - - - - - - - - - - - - - - - - -

Interaktion durch Bewegungen des Körpers oder von Körperteilen.

\textbf{Gestenstile:}
\begin{itemize}
  \item \textbf{Deiktisch:} Zeigen auf Objekte.
  \item \textbf{Manipulativ:} Bewegen oder Verändern von Objekten.
  \item \textbf{Symbolisch:} Posen und kodierte Bewegungen mit festgelegter Bedeutung.
\end{itemize}

% -----------------------------------------
\subsection{Interaktionsumgebungen}
% -----------------------------------------

Interaktionsumgebungen sind die technischen Umgebungen, in denen Aufgaben, Techniken
und Stile umgesetzt werden.

\begin{itemize}
  \item \textbf{CLI (Command Line Interface):} Textbasiert, streng reglementiert
        und kodiert.
  \item \textbf{GUI (Graphical User Interface):} Nutzt Metaphern (z.\,B. den
        „Schreibtisch") und grafische Widgets.
  \item \textbf{WIMP-Systeme:} Unterform der GUI -- bestehend aus
        \textbf{W}indows, \textbf{I}cons, \textbf{M}enus und \textbf{P}ointers.
  \item \textbf{NUI (Natural User Interface):} Ermöglicht „natürliche" Interaktionen,
        z.\,B. durch Berührung (iPad) oder Sprache.
  \item \textbf{TUI (Tangible User Interface):} Greifbare Schnittstellen, bei denen
        physische Objekte als Eingabemedien dienen.
  \item \textbf{BCI (Brain Computer Interface):} Direkte Steuerung durch
        Gehirnströme (Biosignale).
\end{itemize}
